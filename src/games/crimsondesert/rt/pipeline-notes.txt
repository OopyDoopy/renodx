================================================================================
  CRIMSON DESERT — RT DIFFUSE GI PIPELINE
  Reverse-engineered from decompiled SM 6.6 compute shaders
================================================================================


PIPELINE EXECUTION ORDER
========================

  1. GenerateDiffuseRays        [8x8]    Pick a ray direction per pixel
  2. [DXR Hardware Trace]                 Engine-internal DispatchRays
  3. EvaluateDiffuseRadiance     [16x16]  Shade the hit point (full lighting)
  4. TemporalResampleReservoirs  [8x8]    ReSTIR temporal resampling
  5. SpatialResampleReservoirs   [8x8]    ReSTIR spatial resampling
  6. BilateralReconstructionGI   [8x8]    Bilateral upsample of VoxelGI cache

  Running in parallel to the above:
    PrepareDlssRR               [8x8]    Pack G-buffer guides for DLSS-RR
    [DLSS Ray Reconstruction]            Denoises both the RT path and bilateral GI


================================================================================
STAGE-BY-STAGE BREAKDOWN
================================================================================


STAGE 1 — GenerateDiffuseRays (846 lines)
------------------------------------------
Purpose: Generate 1spp ray directions using importance sampling from the
previous frame's tiled radiance cache.

Two paths:

  A) ReSTIR path (when reprojection succeeds):
     - Reprojects current pixel into previous-frame tile space via
       _projToPrevProj matrix.
     - Bilinearly interpolates 4 neighboring tile PDFs from
       g_tiledRadianceCachePlanePrev + g_tiledRadianceCachePDFPrev.
     - Fills 8x8 groupshared arrays:
         _global_1[256] = bilateral-weighted PDFs
         _global_2[192] = octahedron-decoded directions, rotated by
                          _terrainNormalParams.z
     - Bilateral weights use distance^2 Gaussian with depth-plane rejection
       via _diffViewPosAccurate for high-precision position differences.
     - Builds a CDF from the weighted PDFs and samples it with a TEA-hash
       random draw to select a direction.
     - Falls back to cosine hemisphere if total weight < 0.03125.

  B) Cosine fallback (reprojection fails):
     - Uses TEA hash -> MCG multiplier (x * 48271) for white noise.
     - _renderParams.w == 0: uniform hemisphere (PDF = 0.5)
     - otherwise: cosine hemisphere (PDF = 2 * cos_theta)

Outputs:
  g_raytracingHitResultUAV          = float4(dir.xyz, 0)
  g_raytracingDiffuseRayInversePDFUAV = 1 / max(0.01, PDF)


STAGE 2 — [DXR Hardware Trace]
-------------------------------
Not a custom shader. The engine dispatches a DXR DispatchRays that fills:
  - Hit distance (written into g_raytracingHitResultUAV.w)
  - Base color (g_raytracingBaseColor)
  - Normal (g_raytracingNormal)
  - Material stencil (g_materialStencil)


STAGE 3 — EvaluateDiffuseRadiance (2450 lines)
-----------------------------------------------
The monster shader. Shades the RT hit point with the full lighting model.
Major subsystems:

  3a) Surfel radiance lookup at hit point (~lines 400-850):
      - Finds voxel clipmap level for hit position (iterates 8 levels).
      - If level < 6, loads up to 4 surfels from g_surfelIndicesVoxelsTextures
        -> g_surfelDataBuffer.
      - Each surfel contributes weighted radiance based on:
          * Normal agreement (dot product > -0.03125)
          * Proximity (tangential dist^2 < radius^2 * 16)
          * SDF validity check via g_signedDistanceVoxelsTexturesLikeUav
          * Exposure-aware luminance clamping
      - Albedo decoded from packed uint (sRGB -> linear), with BT.2020->709
        color matrix applied.
      - If no explicit albedo from the hit (_226.w == 0), surfel base color
        is used as fallback.
      - Result = indirect bounce lighting at the hit.

  3b) Screen-space reprojection (~lines 850-970):
      - Projects hit point into previous frame via _viewProjRelativePrev.
      - If reprojected point has valid depth (within 5% tolerance):
          * Samples g_sceneColor at that screen position to read previous
            frame's fully-lit color as radiance approximation.
          * Modulated by facing test (ray vs hit normal), distance fade,
            scaled by _renderParams2.x^2.
      - This is a screen-space radiance cache — cheap but limited to
        previously-visible surfaces.

  3c) SDF voxel ray march (~lines 970-2010):
      - When hit distance is within voxel bounds, launches sphere-trace
        through SDF voxel field:
          * g_axisAlignedDistanceTextures (per-axis distance fields)
          * g_signedDistanceVoxelsTexturesLikeUav (SDF sampling)
      - Up to 192 steps marching along ray direction.
      - Accumulates occlusion (0 -> 1) based on SDF distance.
      - At termination, performs second surfel lookup for colored radiance.
      - Result = SDF-traced shadow/bounce lighting with surfel color.

  3d) Blending SDF + screen-space paths (~lines 2010-2070):
      - Distance fade: saturate(5 - linearZ * 0.0195) — screen-space
        fades out by ~256 units.
      - SDF confidence: whether SDF terminated on geometry.
      - Screen-space confidence: whether reprojection succeeded.
      - Weighted blend produces combined hit distance and radiance.

  3e) Full shadow evaluation at blended hit point (~lines 2070-2810):
      Projects the hit point into every shadow system:
        * Static shadow maps (2 cascades) — g_shadowDepthArray +
          g_shadowColorArray, bilinear depth comparison with normal bias
        * Dynamic shadow maps (2 cascades) — g_dynamicShadowDepthArray,
          selects sun or moon based on _sunDirection.y
        * Terrain heightmap shadow — g_terrainShadowDepth (1024x1024,
          GatherRed + bilinear PCF)
        * Cloud volume shadow — g_texCloudVolumeShadow (3D texture)
        * Atmospheric transmittance — Rayleigh + aerosol extinction via
          g_texNetDensity LUT
      Also samples g_sceneShadowColor for precomputed shadow color.

  3f) Specular + diffuse BRDF evaluation (~lines 2810-3210):
      Full Cook-Torrance GGX microfacet BRDF at the hit point:
        * Schlick Fresnel with F0 from albedo * roughness
        * GGX NDF: D = a^4 / (pi * ((NdotH^2 * (a^4-1) + 1)^2))
        * Smith height-correlated geometry term
        * Albedo from surfel data or shadow color array
        * Result scaled by combined shadow * cloud * atmosphere
        * Luminance clamped to exposure * 4096

  3g) Subsurface scattering at origin (~lines 3210-3910):
      Only for stencil 33/55 (skin/foliage materials).
      Re-evaluates the ENTIRE shadow + atmosphere pipeline at the
      shading pixel's own position (not the hit). Computes wrapped
      diffuse: max(0, 0.3 - NdotL) * 0.157 * combinedShadow.
      Reads base color from g_baseColor. Adds SSS contribution.
      This is ~700 lines of nearly-duplicated shadow/atmo work.

  3h) Environment fallback + sky visibility (~lines 3910-4070):
      If hit was very far (> 128 units) and contributed zero radiance:
        * Falls back to g_environmentColor cubemap (MIP 4)
        * Modulated by sky visibility voxels (levels 0-3)
        * Scaled by _renderParams.w
        * [Game Update] Luminance-clamped before directional multiplier:
          raw = cubemap.rgb * skyVisibility
          lum = dot(raw, BT.709)
          clampedLum = min(exposureBudget, lum)
          result = (raw * clampedLum / lum) * directionalMult
          where exposureBudget = exposure3.w * (4096 - roughness * 3584 * terrainNormalParams.z)
          This caps the cubemap probe energy to prevent blowout at dawn/dusk
          when the sky probe is inflated. Rough surfaces get a tighter budget.

  3i) Final output (~lines 4070-4130):
      invPDF = g_raytracingDiffuseRayInversePDF * 2.0
      result = invPDF * (envCubemap * envWeight + renderParams2.y * directLighting)
      if (renderParams.y != 0) result *= cos(theta)
      g_diffuseResultUAV = half4(clamp(exposure4.x * result, 0, 15000), 1 - envWeight)
      g_raytracingHitResultUAV.w = hitDistance (or 1000 if miss)


STAGE 4 — TemporalResampleReservoirsDiffuse (690 lines)
--------------------------------------------------------
Purpose: ReSTIR temporal resampling — accumulate samples across frames.

  - Motion vectors: Dynamic objects (stencil 14-20, 53-67, 105, 98) use
    per-pixel g_velocity buffer. Static geometry uses _projToPrevProj matrix.
  - RenoDX: Area ReSTIR subpixel tracking temporal reuse — compensates
    reprojection coordinates by the inter frame TAA jitter delta
    (_temporalAAJitter.xy - .zw) so temporal lookups account for where the
    previous frame's sample was actually generated.
  - 5 candidate search: 4 motion-reprojected positions with +/-1 pixel
    offsets + 1 at current pixel position (iteration #4).
  - Validation:
      * World-space distance < threshold (0.02-0.1x linearZ, relaxed by jitter)
      * Depth consistency
      * Normal hemisphere check (dot > 0)
  - RIS reservoir merge: target PDF = luminance(radiance) * cos_theta / pi.
    Weighted sampling with previous frame's hit geometry + radiance.
  - Reservoir aging cap:
      * 32 samples when _renderParams.x > 0
      * Otherwise: 256 - saturate(exposure * 10) * 192
        (= 256 in dark scenes, ~64 in bright scenes)
  - Writes: reservoir buffers (hit geometry uint4 + radiance uint2),
    diffuse result, ray hit result, inverse PDF.


STAGE 5 — SpatialResampleReservoirsDiffuse (365 lines)
-------------------------------------------------------
Purpose: ReSTIR spatial resampling — borrow samples from neighbors.

  - Iteration count: max(4, 16/M) neighbors where M = current reservoir
    sample count. More neighbors when fewer temporal samples available.
  - Neighbor radius: 32px when _renderParams.x > 0, else 8px.
  - Validation:
      * Plane-distance test: |dot(N, neighborPos - pos)| > max(0.5, linearZ)
      * Normal consistency: dot(N_cur, N_neighbor) >= 0
  - RenoDX SPMIS path (RT_QUALITY >= 0.5): Area ReSTIR reconnection shift
    mapping replaces raw solid angle Jacobian. Reconnects at the hit point
    using hit surface normal for both cosine terms, clamped to [0, 4] instead
    of [0, 10]. Backface rejection at reconnection vertex (cos_hq < 1e-4).
  - Jacobian correction: solid-angle ratio between neighbor and current
    pixel. Backface test via hit normal.
  - RIS merge: weighted reservoir sampling combining current + neighbor.
  - Output: g_diffuseResultUAV = half4(luminance*weight, ..., 0)


STAGE 6 — BilateralReconstructionDiffuseGITiled (350 lines)
------------------------------------------------------------
Purpose: Bilateral upsample of the VoxelGI tiled radiance cache to
per-pixel resolution. This is the PARALLEL NON-RT PATH that all pixels get.

  - Screen-space jitter: LCG seed = _frameNumber * 73, R2-like sequence,
    range +/-16 pixels, clamped to screen bounds.
  - Bilateral filter: Samples 4 neighboring tiles from
    g_tiledRadianceCachePlane + g_tiledRadianceCacheOctahedron.
    Per-tile selects 1 of 16 octahedron sub-samples via TEA hash into
    _global_0[32] precomputed offset array.
  - Bilateral weights:
      * Gaussian distance falloff: exp2(-0.144/sigma^2 * dist^2)
      * Normal-plane distance penalty (*50)
      * Depth occlusion test
      * Proximity-based weight clamping
  - AO modulation: g_sceneAO * material-dependent factor:
      * 0.2 for stencil 98/96
      * 0.9 for everything else
  - Output: g_textureResultUAV += bilateralGI * aoFactor (additive)
  - Now uses the RR flag (.w = 0.0), routing its output through DLSS Ray
    Reconstruction for temporal denoising alongside the RT diffuse path.


PARALLEL — PrepareDlssRR (219 lines)
-------------------------------------
Purpose: Pack G-buffer data into DLSS Ray Reconstruction guide buffers.

  - Decodes base color RGBA, applies sRGB^2 gamma approximation.
  - BT.2020 -> BT.709 conversion via hardcoded 3x3 matrix.
  - Specular albedo via Schlick-GGX DFG polynomial approximation.
  - Normal + roughness from g_normalRoughnessOpaque.
  - SSS guide: (sceneColor - sceneColorBeforeSSS) * albedoWeights * 0.25
  - Writes 4 UAVs: diffuse albedo, specular albedo, normal+roughness,
    SSS guide.

  [Game Update] Stencil-gated metallic zeroing for DLSS-RR:
    Stencils {11, 12-19, 24, 29, 54, 65-67, 96, 107} now force metallic
    to 0.0 in the specular albedo output, giving these materials a fixed
    dielectric F0 of 0.04. This prevents non-metallic materials (foliage,
    skin, hair, SSS, transparent, decals) from incorrectly contributing
    metallic specular to Ray Reconstruction. Previously the G-buffer
    metallic channel was used directly for all stencils.


================================================================================
ISSUES & LIMITATIONS
================================================================================


CRITICAL
--------

1. WHITE NOISE THROUGHOUT — NO BLUE NOISE OR LOW-DISCREPANCY SEQUENCES

   TEA (Tiny Encryption Algorithm) + MCG (x * 48271) is used everywhere:
   ray generation, spatial neighbor selection, bilateral jitter, octahedron
   sub-sample selection. White noise has no spectral guarantees — it clusters
   and leaves gaps. Blue-noise or R2/Halton sequences would distribute
   samples more uniformly at the same sample count, significantly reducing
   perceived noise especially at 1spp.

2. [RESOLVED] VOXELGI BILATERAL RECONSTRUCTION NOW DENOISED BY DLSS-RR

   The bilateral reconstruction output now uses the RR flag (.w = 0.0),
   routing it through DLSS Ray Reconstruction for temporal denoising.
   This resolves the previous indoor/window shimmer and motion noise
   that resulted from the undenoised bilateral GI path.

3. +/-16 PIXEL JITTER RANGE IN BILATERAL RECONSTRUCTION IS EXCESSIVE

   The jitter formula: 31.999 * frac(R2) - 16 spans +/-16 pixels. At tile
   boundaries, depth discontinuities, or geometry edges, this pulls samples
   from completely different geometry, creating cross-region bleeding
   artifacts. A range of +/-4 tiles would be far safer.


SIGNIFICANT
-----------

4. TEMPORAL RESERVOIR AGE CAP IS EXPOSURE-DEPENDENT

   cap = 256 - saturate(exposure * 10) * 192

   In dark scenes, reservoirs accumulate up to 256 samples (good convergence).
   In bright outdoor scenes, the cap drops to ~64. This means bright scenes
   are noisier by design — precisely the conditions where noise is most
   visible. Rationale is likely responsiveness to lighting changes outdoors,
   but it creates a quality cliff.

5. COSINE HEMISPHERE FALLBACK DURING CAMERA MOTION

   In GenerateDiffuseRays, when reprojection fails (camera whip,
   disocclusion), the shader falls back to cosine/uniform hemisphere
   sampling with white noise. This dramatically increases variance because
   the importance-sampled cached PDF is no longer available. Combined with
   the 1spp budget, this causes visible noise spikes during fast camera
   movement.

6. SURFEL SYSTEM LIMITED TO NEAR CLIPMAP LEVELS

   Hit point surfels: levels 0-5 only (levels 6-7 get zero radiance).
   SDF termination surfels: levels 0-3 only (even more restricted).
   Distant RT hits rely entirely on screen-space reprojection (which
   requires the hit to be visible in the previous frame) or the environment
   cubemap fallback. Distant indirect lighting can be quite dim or missing.

7. SCREEN-SPACE RADIANCE CACHE IS FRAGILE

   Reprojecting the hit point into previous frame's g_sceneColor fails for:
   off-screen hits, newly-exposed surfaces, surfaces behind the camera, and
   any geometry that moved. The fallback (surfel/SDF path) only covers
   near-field. A non-trivial fraction of hits will get neither screen-space
   nor surfel radiance.

8. ONLY 4 SURFELS PER VOXEL CELL ARE EVALUATED

   The surfel loop caps at 4 iterations. If a voxel contains more than 4
   surfels, excess ones are silently ignored. In complex geometry (foliage,
   dense interiors), this under-samples the radiance field and may bias
   the result.


MINOR
-----

9. SPATIAL RESAMPLING RADIUS DISPARITY: 8px vs 32px

   When _renderParams.x == 0, spatial neighbor radius is only 8px.
   In RT mode it jumps to 32px. The 8px radius may be insufficient
   for convergence in the non-RT fallback path.

10. 5-CANDIDATE TEMPORAL SEARCH MAY MISS WITH LARGE DISOCCLUSIONS

    Temporal resampling tries only 4 motion-reprojected positions
    (+/-1 pixel) + 1 at current position. With thin objects, parallax
    edges, or aggressive motion blur, all 5 can fail validation, causing
    complete temporal rejection. The reservoir starts from scratch with
    only the current frame's 1spp sample.

11. LUMINANCE CLAMPING USES DIFFERENT THRESHOLDS PER STAGE

    Surfel radiance:   clamped to exposure * 512
    Direct lighting:   clamped to exposure * 4096
    Environment:       clamped to exposure * 2048
    Final output:      clamped to 15000 (absolute)

    These different clamp levels create subtle energy imbalances — bright
    surfel bounces are suppressed 8x more aggressively than direct hits,
    potentially under-representing bounce light from bright emitters.

12. SSS PATH DUPLICATES ENTIRE SHADOW + ATMOSPHERE EVALUATION

    For stencil 33/55 (skin/foliage), lines 3210-3910 re-evaluate the
    full shadow cascade + terrain shadow + cloud shadow + atmospheric
    transmittance at the origin pixel (not the hit point). This is ~700
    lines of nearly-duplicated work and a significant ALU cost for those
    material types.

13. NO LARGE-SCALE TERRAIN OCCLUSION IN GI — FLAT VALLEYS BEHIND MOUNTAINS

    The GI system has no mechanism for large-scale terrain occlusion.
    When a mountain blocks the sun (e.g. ~16:00 in a valley), the shadow
    system (SceneShadowTiled) may correctly darken the valley floor for
    direct light via terrain heightmap shadows or cascades, but the GI
    pipeline is completely independent and has no knowledge of this.

    The problem manifests as three interacting failures:

    a) EvaluateDiffuseRadiance evaluates full shadow + lighting at RT hit
       points (section 3e). Hit points on the mountain's backface or
       distant terrain may fall outside shadow cascade coverage, so they
       get evaluated as if in direct sunlight — producing high radiance
       that feeds back into the GI.

    b) The cubemap fallback path (section 3h) modulates by sky visibility
       voxels (g_skyVisibilityVoxelsTexturesLikeUav), but these encode
       static geometry occlusion of the sky hemisphere, not dynamic
       sun-angle-dependent terrain shadow. The cubemap at 16:00 is still
       bright in most directions, pumping omnidirectional sky energy into
       the valley regardless of the mountain.

    c) HBAO fades to zero at ~120 units linear depth (see AO section).
       BilateralReconstructDiffuseGITiled applies AO modulation using
       g_sceneAO, which is also zero at distance. This creates a visible
       disconnect: near the player, SSAO provides some darkening; at
       mid-to-far distance, zero AO + full unoccluded GI = glowing
       terrain. The mountain's backface (no shadow, only GI) appears
       brighter than the valley floor near the player (has AO).

    Confirmed by undraw testing:
    - Undrawing GenerateDiffuseRays uniformed the scene by killing all
      RT GI downstream, confirming the incorrect energy comes through
      the RT GI path.
    - Undrawing SceneShadowTiled had no effect on the mountain's
      occlusion, confirming the mountain doesn't cast into shadow maps
      (or the terrain shadow doesn't cover it at that angle).

    This is a fundamental architectural gap — the shadow system and GI
    system operate independently with no shared large-scale occlusion
    data. Potential fix approaches that were explored:

    - Shadow-aware GI attenuation in RenderDiffuseTiled: use shadow
      visibility to attenuate GI at distance where AO has faded out.
      Tested but rejected — the approach is too coarse. Shadow color
      encodes direct-light occlusion which doesn't map cleanly to
      indirect-light occlusion. Shadowed areas that should still receive
      strong indirect bounces (e.g. a sunlit wall bouncing light into a
      shadowed courtyard) get incorrectly darkened. The near/far
      transition also creates visible banding in some camera angles.

    - Extending AO range: would require modifying SceneAOTiled to sample
      at greater distances, but the screen-space nature of HBAO means
      distant samples cover less world area and become unreliable.

    - Injecting terrain shadow into EvaluateDiffuseRadiance hit-point
      evaluation: the most correct approach but requires the terrain
      shadow projection matrix to cover hit point positions, which may
      not be the case for distant hits.

    Currently unresolved. The issue is most visible in valleys with
    large terrain features blocking the sun at oblique angles (~15:00-
    17:00 game time).


================================================================================
RNG DETAILS
================================================================================

All shaders use the same RNG chain:
  1. TEA (Tiny Encryption Algorithm) — 16-round Feistel cipher
     Seeded with (_frameNumber + pixel.x, pixel.y) or similar
  2. MCG multiplier: next = current * 48271
     Used for sequential draws within a shader invocation
  3. reversebits() for Cranley-Patterson rotation (in GenerateDiffuseRays)

There is no spatial or temporal stratification. Every pixel independently
draws from an uncorrelated white noise stream. This is adequate for high
sample counts but suboptimal for the 1spp budget this pipeline operates at.

================================================================================
SPECULAR RADIANCE — ANIMATED GEOMETRY MOTION ARTIFACT
================================================================================

STATUS: Confirmed base game limitation. Not fixable from shader replacement.

SYMPTOM
-------
RT specular on animated geometry (characters, creatures, cloth) appears
brighter/different when the character is in motion (walking, idle sway,
combat animations) compared to when the character is static. Most visible
on high-albedo or metallic surfaces (snow-covered armor, polished metal)
where bright background radiance (sky, snow) bleeds into the specular.

The artifact does NOT occur during camera-only motion with frozen game
time (photo/camera mode), confirming it is tied to skeletal animation
and per-object movement, not camera reprojection.

ROOT CAUSE
----------
EvaluateSpecularRadiance_0x70DA2ED5 has two radiance sources that assume
static geometry, with no compensation for skeletal animation:

1. SCREEN-SPACE RADIANCE CACHE (primary source, ~70% of artifact)
   Variables: _3320-_3322 (assigned from _1236-_1238)
   
   The shader reprojects the RT hit point into previous-frame screen
   space using _viewProjRelativePrev (lines ~1116-1138). This matrix
   only accounts for camera motion — it has no knowledge of per-object
   skeletal animation, cloth simulation, or any mesh deformation.
   
   When the character animates:
   a) Frame N: character's surface is at world position A
   b) Frame N+1: character's surface moved to position B (new pose)
   c) RT ray hits position B, shader reprojects B via _viewProjRelativePrev
   d) The reprojected UV corresponds to where B was in frame N's camera,
      but in frame N the character's surface was at A, not B
   e) The UV lands on whatever was behind position B in frame N (sky, snow)
   f) That bright background radiance is used as specular for the character
   
   The reprojected radiance is sampled from either:
   - g_sceneDiffuseHalfPrev (t164) via _viewProjRelativePrev (preferred path)
   - g_sceneColor (t173) at current-frame projected UV (fallback)
   
   Both sources contain the fully-lit previous frame, so sampling the
   wrong position yields background radiance instead of surface radiance.

2. SURFEL/SDF RADIANCE (secondary source, ~30% of artifact)
   Variables: _3327-_3329 (via _2057-_2059 from _2040-_2042)
   
   The surfel lookup uses the RT hit point's world position to index
   into the voxel clipmap (g_surfelIndicesVoxelsTextures → 
   g_surfelDataBuffer). Surfels are baked/cached static radiance probes
   that don't track animated geometry. When the character moves, the
   hit point samples surfels that belong to the space the character
   just moved into, not the character's actual surface.
   
   The SDF ray march (up to 192 steps through 
   g_signedDistanceVoxelsTexturesLikeUav) similarly operates on the
   static voxel field and doesn't account for animated mesh positions.

INVESTIGATION METHOD
--------------------
Confirmed via inline debug views in EvaluateSpecularRadiance (compile-time
SPECULAR_DEBUG_VIEW define, views 1-21):

- View 15 (_3320-_3322, screen-space only): Shows motion artifact → primary
- View 13 (_3327-_3329, surfel/SDF only): Shows motion artifact → secondary  
- View 20 (kill screen-space, keep surfel): Most artifact gone, residual remains
- View 21 (kill both screen-space and surfel): All motion artifacts eliminated
- View 9 (cubemap only): Clean, no motion dependency
- Camera mode test (frozen game time): No artifact during camera-only motion

WHY IT CANNOT BE FULLY FIXED FROM SHADER REPLACEMENT
-----------------------------------------------------
The fundamental problem is that the RT hit point world position corresponds
to the character's CURRENT frame pose, but the reprojection and surfel
lookup need the PREVIOUS frame pose to be correct. The shader has no access
to:

- Per-vertex skinning matrices (current or previous frame)
- Per-object motion vectors at the hit point
- Previous-frame mesh positions for the hit surface

The DXR hardware trace (engine-internal DispatchRays) fills
g_raytracingHitResult with hit distance only. The hit point is
reconstructed geometrically: hitPos = pixelWorldPos + hitDist * reflDir.
No per-object motion data is carried through from the trace.

The engine does have per-pixel motion vectors (g_velocity) for dynamic
objects, but these are screen-space 2D vectors at the SHADING pixel, not
at the RT HIT point. They describe how the shading pixel moved between
frames, not how the hit surface moved.

APPLIED MITIGATION (DISABLE_SS_SPECULAR)
-----------------------------------------
The screen-space radiance cache path is bypassed entirely by forcing
__defer_1023_1309 = true at the top of the screen-space branch. This
sends all pixels through the surfel/SDF fallback path, which naturally
falls through to the far-hit cubemap (mip 4 + sky visibility voxels)
when surfels produce zero radiance.

This approach was chosen over zeroing the screen-space outputs because:
- Zeroing _3320-_3322 and forcing _3326=0 triggered the mip 0 cubemap
  fallback at _3362, which produced unoccluded, overly bright specular
  with no sky visibility attenuation.
- Skipping the screen-space branch entirely preserves the vanilla energy
  balance and AO behavior of the surfel/SDF/cubemap fallback path.

Trade-off: Pixels that previously got high-quality screen-space specular
(static geometry with valid reprojection) now get the lower-quality
surfel/cubemap path. The surfel path still has a minor (~30%) motion
artifact from static voxel lookups on animated geometry, but this is
much less visible than the screen-space artifact.

Controlled by compile-time define DISABLE_SS_SPECULAR (0=off, 1=on)
at the top of EvaluateSpecularRadiance_0x70DA2ED5.cs_6_6.hlsl.

POTENTIAL ENGINE-LEVEL FIXES (not implementable via mod)
--------------------------------------------------------
1. Output previous-frame hit position from the DXR closest-hit shader
   (requires engine access to previous-frame BVH or skinning matrices)
2. Use per-object motion vectors at the hit point for reprojection
   correction (requires the DXR trace to output object velocity)
3. Detect animated stencils at the hit point and reduce/disable the
   screen-space path for those pixels (partial mitigation, loses quality)
4. Temporal clamping of the screen-space contribution based on
   frame-to-frame luminance consistency (would reduce the artifact
   but also reduce legitimate specular changes)

AFFECTED SHADERS
----------------
- EvaluateSpecularRadiance_0x70DA2ED5.cs_6_6.hlsl (this shader)
- Likely also affects EvaluateDiffuseRadiance variants (same screen-space
  radiance cache architecture, section 3b in pipeline notes), though the
  diffuse path may be less visually obvious due to lower specular contrast.

DEBUG VIEWS (compile-time, SPECULAR_DEBUG_VIEW define)
------------------------------------------------------
Views 1-21 are available for future investigation. Key views:
  1  = Path selector (R=screen-space, G=surfel/SDF, B=cubemap)
  2  = g_sceneColor at this pixel's UV (raw input check)
  3  = g_sceneDiffuseHalfPrev at this pixel's UV (raw input check)
  10 = Final specular pre-exposure (full pipeline output)
  13 = Surfel/SDF radiance only
  15 = Screen-space radiance only (primary artifact source)
  20 = Kill screen-space (confirm primary source)
  21 = Kill screen-space + surfel (confirm both sources)
Set SPECULAR_DEBUG_VIEW to the view number at the top of the shader file
and rebuild with: cmake --build build --target crimsondesert --config Release
